## 🎯 定義書概覽

此定義書定義了企業 HR 自動化系統的核心問題、決策規則和成功指標。

**核心目標**：透過自動化工作流，減少 HR 人員的重複勞動，提高流程透明度，縮短審批時間。

**適用範圍**：中型至大型企業（100-5,000 名員工）的 HR 部門。

**實現方式**：使用 Claude Code Dynamic Workflows 自動生成核心邏輯，由企業根據此定義書修改參數，自行生成符合企業現況的工作流代碼。

## 📊 三個核心 HR 場景

### 場景 1：請假申請與審批工作流

**場景背景**：
企業員工需要申請各種類型的假期（年假、病假、事假、婚假等），經過多級審批後才能確認。該流程涉及複雜的額度檢查、審批規則和通知流程。

**問題定義**：
- 員工手動填寫請假單，經理需要手動審核
- 政策檢查依賴人工記憶和經驗
- 假期額度計算容易出錯
- 審批進度不透明，員工頻繁詢問

**自動化目標**：
1. 自動檢查員工假期額度和年度使用情況
2. 根據公司政策自動計算審批路徑（直屬經理 → 部門主管 → HR）
3. 實時通知相關審批人員
4. 將審批結果自動同步到考勤系統
5. 為員工提供實時的申請狀態查詢

**成功指標**：
| 指標 | 目標值 | 度量方式 |
|------|--------|---------|
| 自動化覆蓋率 | ≥ 95% | （已自動審批的請假單 / 總請假單） × 100% |
| 平均審批時間 | ≤ 4 小時 | 從提交到最終審批完成的平均時間 |
| 異常處理率 | ≤ 5% | （需要人工干預的請假單 / 總數） × 100% |
| 用戶滿意度 | ≥ 4.0/5.0 | 員工滿意度調查得分 |
| 系統可用性 | ≥ 99.5% | 系統正常運行時間 / 總時間 |

**決策規則**：

#### Rule 1：額度檢查
```
IF 請假類型 = "年假" THEN
  檢查員工年度剩餘年假額度
  IF 剩餘額度 ≥ 申請天數 THEN 允許申請
  ELSE 拒絕申請，返回"額度不足"錯誤
END

IF 請假類型 = "病假" THEN
  檢查員工月病假額度（5天/月）
  IF 當月已用天數 + 申請天數 ≤ 5 THEN 允許申請
  ELSE 需要上級經理確認
END

IF 請假類型 = "事假" THEN
  檢查員工年度事假額度（3天/年）
  IF 年度已用天數 + 申請天數 ≤ 3 THEN 允許申請
  ELSE 拒絕申請
END
```

#### Rule 2：審批路徑確定
```
IF 請假天數 ≤ 2 THEN
  審批人 = 直屬經理
  審批級數 = 1

ELSE IF 請假天數 ≤ 5 THEN
  審批人 = [直屬經理, 部門主管]
  審批級數 = 2

ELSE IF 請假天數 > 5 THEN
  審批人 = [直屬經理, 部門主管, HR 負責人]
  審批級數 = 3

END

特殊情況：
IF 請假跨越公司重要會議 THEN
  額外审批人 = 相關項目經理
END
```

#### Rule 3：自動審批決策
```
IF 員工過去12個月內請假記錄 = 優秀（無遲到、無無故缺勤） THEN
  經理直接批准概率 = 95%
  自動批准（無需人工干預）

ELSE IF 員工請假記錄 = 良好 THEN
  經理直接批准概率 = 70%
  需要經理確認

ELSE IF 員工請假記錄 = 一般或較差 THEN
  需要多級審批並確認理由
  自動審批概率 = 30%

END
```

#### Rule 4：通知發送
```
提交申請時 → 通知直屬經理 + HR
審批通過 → 通知員工 + 更新考勤系統 + 通知相關部門
審批拒絕 → 通知員工（含拒絕原因）+ 建議修改方案

IF 申請未在24小時內得到回應 THEN
  自動提醒審批人 (每6小時一次，最多3次)
END
```

**約束條件**：
- 不支持跨越更換部門的請假申請（需要新部門經理確認）
- 年假只能在員工離職前一個月內自動批准
- 病假需要醫療證明（由 HR 人工驗證）
- 申請時間必須晚於當前時間至少 24 小時

### 場景 2：報銷申請與審批工作流

**場景背景**：
員工因工作需要發生各類支出（出差、辦公用品、會議費等），需要提交報銷申請，經過審批後才能獲得報銷。該流程涉及複雜的預算檢查、合規性驗證和財務流程。

**問題定義**：
- 員工手動整理收據並填寫報銷單
- 對於合規性和預算額度的判斷缺乏一致標準
- 審批層級多，週期長（平均 10-15 天）
- 收據丟失或不符合要求導致頻繁返工
- 財務部門手動輸入到記帳系統

**自動化目標**：
1. 自動驗證收據的合規性（金額、發票類型、日期）
2. 根據部門預算自動確定審批層級
3. 將審批結果自動同步到財務系統
4. 為員工提供即時反饋和改進建議
5. 自動生成報銷統計報告（用於預算監控）

**成功指標**：
| 指標 | 目標值 | 度量方式 |
|------|--------|---------|
| 自動化覆蓋率 | ≥ 90% | （自動審批的報銷單 / 總數） × 100% |
| 平均審批時間 | ≤ 5 天 | 從提交到最終審批完成的平均時間 |
| 規則驗證準確率 | ≥ 98% | （系統判斷正確的案例 / 總案例） × 100% |
| 合規性檢查覆蓋率 | 100% | 所有提交的報銷單都被檢查 |
| 報銷成功率 | ≥ 85% | （首次通過審批的報銷單 / 總數） × 100% |

**決策規則**：

#### Rule 1：收據合規性檢查
```
FOR 每一項報銷申請 DO
  IF 發票金額 ≤ 0 OR 發票金額 > 10,000 THEN
    標記為 "金額異常"，需要人工審查
  END

  IF 發票日期 > 當前日期 OR 發票日期 < (當前日期 - 90天) THEN
    標記為 "日期異常"，拒絕報銷
  END

  IF 發票類型 = "普通發票" THEN
    檢查發票號碼格式
    IF 格式正確 THEN 通過
    ELSE 標記為 "發票號無效"，拒絕
  END

  IF 發票類型 = "增值稅發票" THEN
    驗證納稅人識別號
    IF 驗證成功 THEN 通過
    ELSE 標記為 "稅號無效"，拒絕
  END

END
```

#### Rule 2：支出類別和額度限制
```
報銷類別額度限制表：
┌─────────────────────┬────────────┬─────────────┐
│ 報銷類別            │ 單筆限額   │ 月度限額    │
├─────────────────────┼────────────┼─────────────┤
│ 出差住宿             │ 2,000      │ 10,000      │
│ 出差餐飲             │ 200/天     │ 2,000/月    │
│ 交通費               │ 5,000      │ 15,000/月   │
│ 辦公用品             │ 500        │ 2,000/月    │
│ 會議費               │ 1,000      │ 5,000/月    │
│ 培訓費               │ 3,000      │ 10,000/年   │
└─────────────────────┴────────────┴─────────────┘

IF 單筆申請金額 > 該類別單筆限額 THEN
  需要部門經理確認並提供業務理由
END

IF 員工月度累計報銷 > 月度限額 THEN
  標記為 "超額"，轉人工審批
END
```

#### Rule 3：部門預算檢查
```
IF 提交報銷申請 THEN
  部門預算檢查 = {
    年度預算: 部門年度預算額度
    已用額度: SUM(本年度已批准報銷金額)
    剩餘額度: 年度預算 - 已用額度
  }

  IF 申請金額 ≤ 部門剩餘額度 THEN
    預算檢查通過，批准審批
  ELSE
    需要財務部門或部門主管確認
    IF 確認通過 THEN 從其他預算科目轉移
    ELSE 拒絕報銷申請
  END

END
```

#### Rule 4：審批路徑確定
```
IF 申請金額 ≤ 500 THEN
  審批人 = 直屬經理
  審批級數 = 1

ELSE IF 申請金額 ≤ 2,000 THEN
  審批人 = [直屬經理, 部門主管]
  審批級數 = 2

ELSE IF 申請金額 ≤ 5,000 THEN
  審批人 = [直屬經理, 部門主管, 財務經理]
  審批級數 = 3

ELSE IF 申請金額 > 5,000 THEN
  審批人 = [直屬經理, 部門主管, 財務經理, CFO]
  審批級數 = 4

END

特殊情況：
IF 報銷類別 = "培訓費" AND 申請金額 > 2,000 THEN
  額外審批人 = HR 經理（確認培訓合規性）
END
```

#### Rule 5：自動付款指令
```
IF 報銷申請狀態 = "已全部批准" THEN
  生成付款指令（包含：員工姓名、金額、銀行卡號）
  發送至財務系統
  預計付款時間 = 審批完成後 2-3 個工作日
  員工狀態更新 = "待付款"

ELSE IF 報銷申請狀態 = "已拒絕" THEN
  發送通知給員工，說明拒絕原因
  建議員工的改進方案
  允許員工重新提交

END
```

**輸入規格**：
```json

  "requestId": "REIMB-2026-000456",
  "employeeId": "E12345",
  "department": "銷售部",
  "submitDate": "2026-06-04",
  "expenses": [
    {
      "category": "出差住宿",
      "description": "北京出差住宿（2026-06-01 至 2026-06-03）",
```

**約束條件**：
- 報銷申請必須在發生支出後 90 天內提交
- 每份報銷申請必須至少包含一張有效發票
- 超過 5,000 元的報銷申請需要附加業務證明文件
- 員工不能對自己的報銷申請進行審批

### 場景 3：培訓申請與管理工作流

**場景背景**：
企業鼓勵員工進行職業發展和培訓，需要管理培訓申請、預算、合規性和效果評估。該流程涉及複雜的資格檢查、預算管理、合規性驗證和效果追蹤。

**問題定義**：
- 員工對培訓政策了解不足，導致申請不符合要求
- 培訓預算分配和跟蹤缺乏透明度
- 人工審批流程複雜，審批時間長（平均 7-10 天）
- 無法跟蹤培訓效果和員工學習進度
- 缺乏數據支持培訓政策的優化

**自動化目標**：
1. 自動檢查員工的培訓資格（薪級、部門、過往培訓記錄）
2. 根據培訓內容和員工級別自動預批或轉人工審批
3. 監控部門和公司培訓預算的使用情況
4. 自動向員工發送培訓提醒和安排通知
5. 收集和追蹤培訓效果反饋

**成功指標**：
| 指標 | 目標值 | 度量方式 |
|------|--------|---------|
| 自動化覆蓋率 | ≥ 85% | （自動審批的培訓申請 / 總數） × 100% |
| 平均審批時間 | ≤ 3 天 | 從提交到最終審批完成的平均時間 |
| 員工培訓參與率 | ≥ 70% | （完成培訓的員工數 / 總員工數） × 100% |
| 培訓滿意度 | ≥ 4.0/5.0 | 員工培訓滿意度調查得分 |
| 預算控制精度 | ±5% | 實際花費 vs 計劃預算的偏差 |

**決策規則**：

#### Rule 1：員工資格檢查
```
IF 申請培訓 THEN
  檢查員工資格：

  基礎資格：
    - 員工在公司工作滿 3 個月
    - 過去 6 個月績效評分 ≥ 3.0/5.0
    - 未在試用期內

  IF 不符合基礎資格 THEN
    拒絕申請，告知員工原因和改進建議
    RETURN
  END

  薪級限制：
    IF 薪級 ∈ [P1, P2, P3] THEN  // 初級員工
      允許的培訓類別 = [通用技能, 行業基礎, 軟技能]
      年度培訓預算 = 2,000 元

    ELSE IF 薪級 ∈ [P4, P5, P6] THEN  // 中級員工
      允許的培訓類別 = [通用技能, 專業技能, 領導力基礎, 行業進階]
      年度培訓預算 = 5,000 元

    ELSE IF 薪級 ∈ [P7, P8, P9] THEN  // 高級員工
      允許的培訓類別 = [所有類別]
      年度培訓預算 = 10,000 元
    END

  IF 申請培訓類別 NOT IN 允許的培訓類別 THEN
    拒絕申請，建議員工選擇符合薪級的培訓
  END

END
```

#### Rule 2：培訓費用限制
```
培訓費用限制表：
┌─────────────────────────┬──────────────┬──────────────────┐
│ 培訓類型                │ 課程時長     │ 單次費用限額      │
├─────────────────────────┼──────────────┼──────────────────┤
│ 線上課程（自主式）       │ 任意         │ ≤ 500            │
│ 線上課程（直播式）       │ ≤ 40小時     │ ≤ 1,000          │
│ 線上課程（直播式）       │ > 40小時     │ ≤ 2,000          │
│ 線下課程（短期）         │ ≤ 5天        │ ≤ 2,000          │
│ 線下課程（中期）         │ 5-30天       │ ≤ 5,000          │
│ 線下課程（長期）         │ > 30天       │ ≤ 10,000         │
│ 學位教育（本科/碩士）    │ 2-3年        │ ≤ 30,000/年      │
│ 專業認證考試             │ 任意         │ ≤ 3,000          │
└─────────────────────────┴──────────────┴──────────────────┘

IF 申請培訓費用 > 限額 THEN
  需要部門主管和HR確認理由
END
```

#### Rule 3：年度培訓預算檢查
```
年度培訓預算檢查：

年度個人預算 = 員工薪級對應的年度預算（見 Rule 1）
已用預算 = SUM(已批准完成的培訓費用)
剩餘預算 = 年度預算 - 已用預算

IF 申請金額 ≤ 剩餘預算 THEN
  預算檢查通過
ELSE
  超過個人年度預算上限，需要部門或HR確認是否從部門預算撥款
END
```

#### Rule 4：審批路徑確定
```
IF 申請培訓類別 = "通用技能" AND 申請費用 ≤ 500 THEN
  可以自批（無需經理審批）
  自動批准

ELSE IF 申請培訓類別 = "通用技能" AND 申請費用 ≤ 2,000 THEN
  審批人 = 直屬經理

ELSE IF 申請培訓類別 = "專業技能" OR 申請費用 > 2,000 THEN
  審批人 = [直屬經理, HR 經理]

ELSE IF 申請培訓類別 = "領導力" AND 申請費用 > 3,000 THEN
  審批人 = [直屬經理, HR 經理, 部門主管]

END
```

#### Rule 5：自動提醒和追蹤
```
IF 培訓申請被批准 THEN
  1. 發送確認郵件給員工（包含課程信息、報名截止日期）
  2. 提醒員工完成報名
  3. 在培訓開始前 7 天自動發送日程提醒
  4. 在培訓開始前 1 天發送最終提醒

IF 員工參加培訓 THEN
  1. 發送簽到提醒（培訓開始時）
  2. 在培訓結束後自動發送滿意度調查
  3. 在結束後 14 天進行效果追蹤

IF 培訓結束 30 天後 THEN
  1. 提醒員工分享學習成果（在團隊會議上展示或撰寫報告）
  2. 更新員工發展檔案
  3. 為經理提供員工學習進度報告

END
```

## 🔄 流程異常和降級策略

### 異常情況處理

| 異常情況 | 觸發條件 | 處理方案 |
|---------|---------|---------|
| 系統故障 | 自動化工作流不可用 | 自動切換到人工審批流程 |
| 審批超時 | 審批人超過 24 小時未回應 | 自動升級到上級領導 |
| 數據異常 | 員工信息不一致 | 人工驗證後修正數據 |
| 審批衝突 | 多個審批人意見不一致 | 提交到 HR 最終判決 |
| 預算異常 | 預算計算出現偏差 | 財務部門人工核對 |

### 升級/降級策略

```
升級（當自動化無法處理時）：
  自動化工作流 → 人工審批 → 專家決策 → 政策修訂

降級（當系統出現故障時）：
  精簡自動化 → 人工為主 + 系統輔助 → 完全人工流程
```
