## 📚 三層學習路徑

### 層級 1：理解問題定義（15 分鐘）

📖 讀：`計劃1_企業HR自動化定義書.md`

- 企業 HR 自動化的「3 個核心場景」（請假、報銷、培訓）
- 每個場景的自動化目標和決策規則
- 量化的成功指標

**核心問題**：
> 企業 HR 部門的痛點是什麼？如何用系統化的方式解決？

### 層級 2：學習生成方法（30 分鐘）

📋 步驟：
1. 打開 `計劃1_企業HR自動化定義書.md`
2. **修改成你的企業場景**（調整額度、審批層級、規則）
3. **提交給 Claude Code**：
   ```
   「根據這份定義書，用 Dynamic Workflows 自動生成 HR Workflow。」
   ```
4. Claude Code 自動生成代碼 + 邏輯驗證

**關鍵技能**：
> 如何定義問題，讓 AI 自動生成解決方案？

### 層級 3：執行和迭代（1 小時）

✅ 循環：執行 → 驗證 → 改進 → 分享

1. 在企業中運行生成的工作流
2. 收集反饋和執行數據
3. 改進定義書（調整不合適的規則）
4. 重新生成代碼 → 獲得更好的結果
5. 分享改進的定義書給社群

**持續優化**：
> 每一次迭代，定義書變得更完善，生成的代碼更符合企業現況。

### Dynamic Workflows 是什麼？

Claude Code 的獨特能力，可以：
1. 讀取「問題定義書」（Markdown）
2. 理解決策規則和成功指標
3. 自動生成完整的可執行工作流代碼
4. 支持多種語言（JavaScript / Python / Go / ...）
5. 生成的代碼是可複用的資產

### 為什麼這很重要？

```
傳統方法：代碼 → 修改 → 測試 → 部署（7-10 天）
DW 方法： 定義書 → 自動生成 → 測試 → 部署（2-3 天）

節省時間：60-70%
節省成本：50-60%
```

## ❓ 常見問題

### Q1：定義書和代碼的區別是什麼？

```
定義書 = 永恆的（描述問題本質）
代碼  = 易變的（實現細節可隨時改變）

例：
定義書：「年假每位員工 15 天，經理可以批准 ≤ 2 天的請假」
代碼1：JavaScript 實現
代碼2：Python 實現
代碼3：Go 實現
...

三種代碼都會隨著技術演進而改變
但定義書可能 5 年都不變（除非公司政策改變）
```

### Q2：我可以修改定義書嗎？

**完全可以！** 修改定義書是整個系統的核心：

```
步驟 1：下載定義書
步驟 2：根據企業需求修改
        例：改額度、加規則、調審批層級
步驟 3：提交給 Claude Code
步驟 4：自動生成新代碼
步驟 5：試運行
```

### Q3：生成的代碼可靠嗎？

```
✅ 定義書 + Claude Code 生成的代碼具有：
   - 邏輯清晰（基於明確的決策規則）
   - 可驗證（對照定義書檢查）
   - 可審計（代碼有完整的決策歷史）

📋 建議流程：
   1. 生成代碼
   2. 代碼審查（IR 驗證邏輯是否符合定義書）
   3. 測試（用真實數據測試）
   4. 小范圍試運行
   5. 全量上線
```

### Q4：為什麼不直接提供現成的代碼？

```
舊方法：給你現成代碼
  ❌ 企業差異大，改代碼難度高
  ❌ 每個企業都要改，成本高
  ❌ 改出 Bug，責任不清
  ❌ 複用性低

新方法（DW）：給你定義書
  ✅ 定義書是通用的（修改參數就夠了）
  ✅ 任何人都能理解和修改（不用懂代碼）
  ✅ 自動生成，準確性高
  ✅ 複用性高（定義書 → 多語言實現）
```

### Q5：如何確保企業數據安全？

```
數據安全原則：
1. 定義書只定義規則，不存儲企業數據
2. 代碼運行在企業自己的環境中
3. 與 Google Workspace / 釘釘 等集成時，
   使用官方 API，遵守安全規範
4. 完整的審計日誌記錄每個決策

推薦部署方式：
   企業內網 → Claude Code DW → 企業系統
   （所有數據保留在企業內部）
```

```
   → 應用到實際場景
   → 用 Claude Code 生成代碼

5️⃣ 分享：在社群中發布你的定義書（正進行中）
   → 幫助其他企業
   → 獲得反饋和改進建議
```
