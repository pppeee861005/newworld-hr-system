## 📚 前置知識（5 分鐘）

### 什麼是定義書？

**定義書** = 一份 Markdown 文檔，描述：
- 業務問題是什麼？
- 要自動化哪些決策？
- 成功的標準是什麼？

**不是代碼**，不需要懂 JavaScript / Python / 等等。

### 什麼是自動生成？

Claude Code 會：
1. 讀取你的定義書
2. 理解其中的決策規則
3. 自動寫出可執行的代碼（JavaScript / Python / ...）
4. 代碼邏輯直接對應定義書中的規則

### 為什麼要自動生成？

| 傳統方法 | DW 自動生成 |
|---------|----------|
| HR 定義問題 | HR 定義問題 ✓ |
| 開發者寫代碼 | Claude Code 自動寫代碼 ✓ |
| 反覆溝通調整 | 直接改定義書重新生成 ✓ |
| **7-10 天** | **2-3 天** |
| **容易出錯** | **錯誤少** |
| **成本高** | **成本低** |

## 🚀 5 步快速教學

### Step 1：準備定義書（5 分鐘）

#### 方式 A：使用現成定義書（推薦新手）

```
1️⃣ 下載現成的定義書
   → 計劃1_企業HR自動化定義書.md

2️⃣ 用 Google Docs / Word 打開
   或在 GitHub 上直接查看

3️⃣ 瀏覽所有內容，熟悉結構
   → 3 個場景
   → 每個場景的決策規則
   → 輸入/輸出規格
```

#### 方式 B：自己創建定義書（進階）

如果現成定義書不符合你的企業需求，你可以自己創建。

**定義書的最小結構**：

```markdown
# [項目名稱]

## 場景描述
[這個工作流要解決什麼問題？]

## 成功指標
- 指標 1：目標值
- 指標 2：目標值
- 指標 3：目標值

## 決策規則

### Rule 1：[規則名稱]
```
IF 條件1 THEN
  動作1
ELSE
  動作2
END
```

### Rule 2：[規則名稱]
...

## 輸入規格
```json
{
  "field1": "說明",
  "field2": "說明"
}
```

## 輸出規格
```json
{
  "result1": "說明",
  "result2": "說明"
}
```

## 約束條件
- 約束 1
- 約束 2
- 約束 3
```

**創建 Tip**：
- 盡量詳細（細節越多，代碼越準確）
- 用人類語言（不需要技術術語）
- 包含具體例子（幫助 Claude Code 理解）
- 說明邊界情況（異常、特殊條件）

### Step 2：根據企業需求修改定義書（10 分鐘）

#### 示例 1：修改額度

**原定義書**：
```
年假額度：15 天/年
審批層級：≤ 2 天單人批准，> 2 天雙人批准
```

**你的企業情況**：
```
我們的年假是 20 天，審批都由經理單人批准
```

**修改方法**：
```markdown
### Rule 1：額度檢查
IF 請假類型 = "年假" THEN
  檢查員工年度剩餘年假額度
  IF 剩餘額度 ≥ 申請天數 THEN 允許申請

📝 修改：20 天（原本是 15 天）
```

#### 示例 2：修改審批層級

**原定義書**：
```
IF 申請金額 ≤ 500 THEN
  審批人 = 直屬經理
ELSE IF 申請金額 ≤ 2,000 THEN
  審批人 = [直屬經理, 部門主管]
```

**你的企業情況**：
```
我們規模更大，所有報銷都需要部門主管審批
```

**修改方法**：
```markdown
IF 申請金額 ≤ 500 THEN
  審批人 = [直屬經理, 部門主管]

📝 修改：所有申請都加上部門主管
```

#### 示例 3：增加新規則

**原定義書**：
```
[其他規則...]
```

**你的企業情況**：
```
我們有一個特殊政策：跨越公司休息日的請假需要額外審批
```

**修改方法**：
```markdown
### Rule 5：特殊日期檢查（新增）
IF 請假跨越公司法定假日 THEN
  需要 HR 最終確認
ELSE
  按正常流程
END
```

#### 修改檢查清單

修改前，確保你：

- [ ] 理解了所有決策規則的含義
- [ ] 與相關部門（HR、財務、培訓等）確認了所有修改
- [ ] 在定義書中清楚地標註了修改部分（用 `📝 修改：` 或 `[企業A 特定]` 等標記）
- [ ] 檢查了修改是否會造成規則衝突
- [ ] 保存了修改後的版本（用日期命名，如 `定義書_2026-06-04_CompanyA.md`）

### Step 4：審查和測試生成的代碼（10 分鐘）

#### 第一步：代碼審查

生成的代碼出現後，檢查以下項目：

**檢查清單**：

- [ ] **規則實現** — 每條決策規則都被正確實現了嗎？
  ```
  定義書：Rule 1 - 額度檢查
  代碼：是否有 checkBalance() 函數？
       邏輯是否正確？
  ```

- [ ] **輸入驗證** — 代碼是否驗證了所有必需的輸入字段？
  ```
  定義書：輸入包括 employeeId, leaveType, dayCount
  代碼：是否檢查了這些字段？
  ```

- [ ] **異常處理** — 代碼是否處理了邊界情況？
  ```
  例：員工額度不足 → 返回明確的錯誤信息
      申請日期不合法 → 拒絕並說明原因
  ```

- [ ] **輸出格式** — 輸出是否符合定義書中的規格？
  ```
  定義書：輸出包括 status, approvalPath, approvalHistory
  代碼：是否返回了所有這些字段？
  ```

#### 第二步：邏輯驗證

**用具體例子測試代碼邏輯**：

```javascript
// 測試 Case 1：正常情況
輸入：{
  employeeId: "E001",
  leaveType: "年假",
  dayCount: 3,
  startDate: "2026-06-10",
  employeeRecord: "excellent" // 記錄優秀
}

預期輸出：{
  status: "approved",
  approvalPath: ["DirectManager"],
  approvalHistory: [
    {
      approver: "Manager A",
      decision: "approved",
      reason: "automatically approved for excellent employee"
    }
  ]
}

驗證：代碼是否返回了正確的結果？

// 測試 Case 2：額度不足
輸入：{
  employeeId: "E002",
  leaveType: "年假",
  dayCount: 20, // 超過年假額度
  startDate: "2026-06-10"
}

預期輸出：{
  status: "rejected",
  reason: "insufficient annual leave balance"
}

驗證：代碼是否正確拒絕了申請？

// 測試 Case 3：需要多級審批
輸入：{
  employeeId: "E003",
  leaveType: "年假",
  dayCount: 10,
  startDate: "2026-06-10",
  employeeRecord: "fair" // 記錄一般
}

預期輸出：{
  status: "pending",
  approvalPath: ["DirectManager", "DepartmentHead"],
  // 需要 2 級審批
}

驗證：代碼是否正確設置了審批路徑？
```

## 💡 進階技巧

### 技巧 1：參數化設計（易於修改）

```markdown
## 配置參數（企業自定義）

### 假期額度配置
```javascript
// 每個企業可以自己改這些數字
const LEAVE_CONFIG = {
  ANNUAL_LEAVE: 15,      // 年假天數，改成 20 就是 20 天
  SICK_LEAVE_PER_MONTH: 5,
  PERSONAL_LEAVE_PER_YEAR: 3
}

// 審批層級配置
const APPROVAL_LEVELS = {
  LEVEL_1: { maxDays: 2, approver: "DirectManager" },
  LEVEL_2: { maxDays: 5, approver: ["DirectManager", "DepartmentHead"] },
  LEVEL_3: { maxDays: Infinity, approver: ["DirectManager", "DepartmentHead", "HR"] }
}
```

**優勢**：
- 企業可以在 1 分鐘內調整配置
- 不需要改代碼，只需改數字
- 降低出錯風險
```

### 技巧 2：決策樹可視化

```
用流程圖表示決策規則，幫助理解：

請假申請提交
    ├─ 額度檢查
    │  ├─ ✓ 通過 → 繼續
    │  └─ ✗ 失敗 → 拒絕
    │
    ├─ 審批路徑確定
    │  ├─ ≤ 2天 → Level 1
    │  ├─ ≤ 5天 → Level 2
    │  └─ > 5天 → Level 3
    │
    ├─ 自動批准檢查
    │  ├─ 記錄優秀 + Level 1 → 自動批准
    │  └─ 其他情況 → 等待經理審批
    │
    └─ 通知和同步
       ├─ 已批准 → 同步考勤系統 + 發通知
       └─ 待審批 → 通知審批人
```

## ❓ 常見問題

### Q1：如果生成的代碼有 Bug 怎麼辦？

**A**：有幾種方式解決：

```
方式 1：改進定義書
  假如 Bug 是因為規則理解有誤
  → 在定義書中明確規則
  → 重新生成代碼
  → 這次應該就沒有 Bug 了

方式 2：要求 Claude Code 修改代碼
  「生成的代碼中，[具體 Bug 描述]
   正確的邏輯應該是 [期望的行為]
   請修改。」
  → Claude Code 會修正代碼

方式 3：手動修改代碼
  如果只是小改動（改個參數、加個註解），
  你可以直接修改生成的代碼。
  對於業務邏輯的改動，建議改定義書後重新生成。
```

### Q2：生成的代碼可以商用嗎？

**A**：完全可以。

```
定義書 + Claude Code 生成的代碼都是：
✅ 歸你所有
✅ 可以商用
✅ 可以修改
✅ 可以開源或閉源

沒有任何限制。
```

### Q3：如何確保多家企業使用同一定義書時邏輯一致？

**A**：參數化設計是關鍵。

```
好的設計方式：

定義書 = 通用邏輯 + 參數化配置

例：
```
年假額度應該 >= 申請天數

企業 A：15 天年假
企業 B：20 天年假
企業 C：12 天年假

邏輯都一樣（年假額度檢查）
但參數不同（15 vs 20 vs 12）

→ 同一份定義書，不同的配置
→ 自動生成的代碼邏輯一致
```

### Q4：定義書需要多詳細？

**A**：越詳細越好，但也要平衡。

```
最小：包含決策規則的偽代碼
中等：包含具體場景和異常處理
詳細：包含完整的輸入輸出、約束條件、測試用例

推薦：從中等級別開始
- 足夠清晰讓 Claude Code 理解
- 不過度設計
- 根據使用經驗逐步完善
```

### Q5：可以為同一個場景生成多個版本的代碼嗎？

**A**：完全可以。

```
同一份定義書，不同的代碼實現：

定義書 → 規則邏輯

JavaScript 實現 → 用於 Google Apps Script
Python 實現 → 用於 Django/FastAPI
Go 實現 → 用於高性能服務
...

所有實現都基於同一份定義書
→ 邏輯一致，實現多樣
```

## 🎓 完整案例演練

### 案例：某企業的報銷工作流生成過程

#### 第一步：準備定義書

企業 HR 說：「我們的報銷政策是：
- 單筆報銷 ≤ 500 元由經理批准
- 500-2000 元需要部門主管確認
- > 2000 元需要財務部門審批
- 必須在支出後 90 天內申請」

#### 第二步：修改定義書
